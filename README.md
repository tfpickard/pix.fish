# pix.fish

A personal image gallery with AI enrichment. Single owner (Tom) uploads; the rest of the world can view, react, and comment. On each upload, an image is stored in Vercel Blob, EXIF and a color palette are extracted, three captions, three descriptions, and a tag set are generated by a vision model, and a caption embedding is written for semantic search. One caption becomes the URL slug; the others rotate on display.

The full multi-phase product spec is in [SPEC.md](./SPEC.md). Implementation context for Claude Code is in [CLAUDE.md](./CLAUDE.md).

## What's in the box

| Capability | Status |
|---|---|
| Upload + AI captions/descriptions/tags | shipped |
| EXIF + palette extraction (no `sharp`) | shipped |
| Owner delete, manual caption override | shipped |
| Caption embeddings + semantic search (`/api/search`) | shipped |
| "More like this" neighbors | shipped |
| Anonymous reactions, comments (with moderation), reports | endpoints + schema shipped |
| Async enrichment queue, WebSocket progress, UMAP map | not yet (Phase 4+) |
| Webhooks, runtime AI provider routing UI | not yet (Phase 4+) |

## Stack

- **Next.js 14** (App Router) + TypeScript + Tailwind + shadcn/ui
- **Neon Postgres** with `pgvector`, via `@vercel/postgres` + Drizzle ORM
- **Vercel Blob** for image storage
- **Auth.js v5** -- GitHub OAuth, single-owner allowlist via `OWNER_GITHUB_ID`
- **Anthropic** (`claude-sonnet-4-6`) for captions/descriptions/tags by default; **OpenAI** (`gpt-4o`) wired as an alternate provider, and `text-embedding-3-small` for caption embeddings
- **`exifr`** for EXIF, **`node-vibrant`** for palette (both pure-JS -- avoids `sharp`)
- **bun** as package manager and script runner

## Local development

### 1. Prerequisites

```sh
bun --version    # >= 1.3
node --version   # >= 20 (Next.js still runs on Node under the hood)
```

### 2. Install

```sh
bun install
```

### 3. Environment variables

Copy `.env.example` to `.env.local` and fill in:

| Var | Required | Notes |
|---|---|---|
| `POSTGRES_URL` | yes | Pooled or unpooled Neon connection string. Grab from Neon or the Vercel Postgres integration. |
| `BLOB_READ_WRITE_TOKEN` | yes | Vercel dashboard -> Storage -> Blob -> create store -> Tokens. |
| `AUTH_SECRET` | yes | `openssl rand -hex 32` |
| `AUTH_URL` | yes | `http://localhost:3000` for local dev. |
| `AUTH_GITHUB_ID`, `AUTH_GITHUB_SECRET` | yes | Create a GitHub OAuth app (Settings -> Developer settings). Homepage `http://localhost:3000`, callback `http://localhost:3000/api/auth/callback/github`. |
| `OWNER_GITHUB_ID` | yes | Your numeric GitHub user id. Look it up at `https://api.github.com/users/<username>`. Only this user can upload, edit, delete, or moderate. |
| `ANTHROPIC_API_KEY` | yes | Captions, descriptions, tags. |
| `OPENAI_API_KEY` | yes | Caption embeddings (`text-embedding-3-small`). Without it, uploads still succeed but embeddings stay null and search is empty. |

### 4. Database

```sh
bun run db:push     # drizzle-kit push -- applies the schema to Neon
bun run db:seed     # seeds 3 prompt templates and ~120 taxonomy tags (idempotent)
```

The migration under `drizzle/0000_init.sql` enables the `vector` extension before any tables are created. If you ever wipe the DB and re-push schema, run `bun scripts/ensure-pgvector.ts` first.

### 5. Run

```sh
bun run dev
```

Or use the `dev.sh` wrapper, which env-checks, manages a backgrounded process, and waits for HTTP 200 before returning -- handy when you want a server running across multiple terminal commands:

```sh
./dev.sh -vv start     # env-checked start, waits for http 200
./dev.sh status        # is it running? on what pid?
./dev.sh logs          # tail the dev log
./dev.sh restart       # stop then start
./dev.sh stop          # SIGTERM, then SIGKILL if needed
```

Open `http://localhost:3000`, click "sign in", authorize the GitHub OAuth app, and if you're the owner you'll see admin nav links.

### 6. Upload

Drop an image on `/admin/upload`, optionally add a manual caption, click upload. The route runs EXIF + palette extraction, then 3 parallel vision calls (captions, descriptions, tags), then writes a caption embedding. Expect 15-40 seconds per upload; `maxDuration = 60` is set on the route.

### 7. Backfill embeddings (optional)

If you imported images before embeddings shipped, or `OPENAI_API_KEY` was missing during some uploads:

```sh
bun scripts/backfill-embeddings.ts
```

Idempotent; uses `onConflictDoUpdate` on `(image_id, kind='caption')`.

## Deployment

1. **Vercel project**: import this repo. Framework preset: Next.js. Build command `bun run build`. Install command `bun install`.
2. **Neon Postgres**: add via the Vercel integration. It populates `POSTGRES_URL` and related env vars automatically. Run `bun run db:push` and `bun run db:seed` against the production database (or branch).
3. **Vercel Blob**: Dashboard -> Storage -> Create Blob Store. Link it to the project; `BLOB_READ_WRITE_TOKEN` is injected automatically.
4. **GitHub OAuth (prod)**: create a second OAuth app with callback `https://<your-domain>/api/auth/callback/github`. Set `AUTH_GITHUB_ID`, `AUTH_GITHUB_SECRET`, and `AUTH_URL` in Vercel env.
5. **Secrets**: set `AUTH_SECRET`, `OWNER_GITHUB_ID`, `ANTHROPIC_API_KEY`, `OPENAI_API_KEY` in Vercel env.
6. **Route timeouts**: `POST /api/images` uses `maxDuration = 60`. Hobby tier limits regular routes to 10s; you'll need Fluid Compute enabled (default on new projects) or Pro tier for sync enrichment to complete. If you're on Hobby without Fluid, the upload will time out.

## API surface

Public reads are unauthenticated. Owner-only writes are gated by both `middleware.ts` and an in-handler `isOwner()` check.

| Method | Path | Who | Purpose |
|---|---|---|---|
| `GET` | `/api/images` | public | paginated list |
| `POST` | `/api/images` | owner | upload + enrich |
| `GET` | `/api/images/[slug]` | public | image detail |
| `PATCH` | `/api/images/[slug]` | owner | edit captions/manual caption |
| `DELETE` | `/api/images/[slug]` | owner | delete image + cascade rows + blob |
| `POST` | `/api/images/[slug]/reactions` | public | up/down (toggle by re-post) |
| `POST` | `/api/images/[slug]/comments` | public | submit comment (lands as `pending`) |
| `PATCH` | `/api/comments/[id]` | owner | approve/reject comment |
| `DELETE` | `/api/comments/[id]` | owner | hard-delete comment |
| `POST` | `/api/reports` | public | report image or comment |
| `GET` | `/api/search?q=...` | public | semantic search over captions |
| `*` | `/api/prompts`, `/api/taxonomy`, `/api/api-keys` | owner | admin CRUD (gated by in-handler `isOwner`, NOT middleware) |

## Current limitations

- **Synchronous enrichment.** Each upload blocks the POST for 15-40 seconds. A future phase moves this to a queue with WebSocket progress streaming.
- **Tag filter uses AND semantics only.** OR-toggle is planned.
- **No prompt/taxonomy/keys admin UI yet.** The CRUD endpoints exist (`/api/prompts`, `/api/taxonomy`, `/api/api-keys`); the `/admin/*` UIs to drive them are not all wired.
- **Image dimensions not stored.** The `width`/`height` columns on `images` exist but are always null -- nothing probes them. Add `image-meta.ts` support if you need them.
- **No CLIP / multimodal embeddings.** Only caption-text embeddings are written today. The `embeddings.kind` column reserves `image` and `combined` for later.

## Judgment calls worth knowing

The full plan lives in [`SPEC.md`](./SPEC.md); the short list of decisions that affect day-to-day work:

1. **Combined JSON per field.** One vision call returns all 3 caption variants (and same for descriptions). Cheaper, faster; the `{{variant_number}}` placeholder is wired but currently unused.
2. **Manual caption is variant 4.** Stored as `variant=4, locked=true, isSlugSource=true`, NOT as a replacement of one of the AI variants.
3. **Random caption on display is server-side per request.** No stability across refreshes; intentional. Don't add client-side caching that fights this.
4. **`pgvector` extension is enabled in migration 0.** Ensures embeddings work the moment the table exists. Don't strip the `CREATE EXTENSION` from `0000_init.sql`.
5. **No native image processing.** Pure-JS only (`exifr`, `node-vibrant`). Anything requiring `sharp` or libvips needs to find another way.
6. **GPS EXIF is stripped before persisting.** `images.exif` is returned by public endpoints, so persisting GPS would publish GPS. See `src/lib/image-meta.ts:17-21`.
7. **Embeddings are best-effort, post-transaction.** A failed OpenAI call does not fail the upload; the row stays valid and `scripts/backfill-embeddings.ts` can fill it in later.
8. **No em dashes anywhere.** Use `--` (two hyphens). Project-wide style rule from the bootstrap prompt in SPEC.md.
